## Development Policies — Non-Negotiable

These policies apply to every commit on every ticket without exception.

### No Hardcoded Test Data

All test inputs are generated by a data synthesizer.

- Python: use the `faker` library. Seed with a fixed integer for reproducibility.
- C#: use the `Bogus` library. Seed with a fixed integer for reproducibility.

Synthesizer code is committed. Generated output is never committed. The same seed must produce
the same data on every run on every machine. Tests that produce different results on different
runs are not acceptable.

### No Infrastructure Mocks in Integration Tests

Integration tests use real infrastructure running in Docker. SQL Server 2022, Redis 7, and
Azurite are provisioned via Docker Compose. Do not replace them with in-memory fakes in
integration tests. The `InMemoryEventBusProvider` and `LocalFileSystem` blob provider exist
for unit-test isolation only.

### HTTP Stubs Are Allowed for External Paid APIs Only

Azure OpenAI, Azure Document Intelligence, SendGrid, Microsoft Graph, GhostDraft, PCAS, and the
UE Rating Engine are never called in CI. Stub them at the HTTP transport layer using `respx`
(Python) or `WireMock.Net` (C#). This is isolation of a network boundary, not a mock of
business logic.

### Test on Every Change

If you modify previously implemented code, run the full test suite for all affected components
before committing. A commit that breaks existing tests is not acceptable under any circumstance.

### Atomic Commits

One logical change per commit. One commit equals exactly one of:
- One interface or data model file created, with nothing else
- One data synthesizer created
- One provider implementation file, with nothing else
- One test file, with nothing else
- One bug fix and its corresponding test
- One status update in `implementation-plan.md`

Never bundle unrelated changes. Never squash a day's work into one commit.

### Push After Every Commit

Every commit must be pushed to origin immediately after it is created.

  git add <specific files>
  git commit -m "..."
  git push

Never accumulate local commits. Never push multiple commits at once. The architect reads each
commit as it lands on GitHub. A batch push is opaque and unauditable.

Never use `--force` push. If a push is rejected, investigate why before taking any other action.

### Conventional Commit Format

Every commit message follows this format exactly:

  <type>(kspl-<NNN>): <imperative description under 72 chars>

Types: `feat`, `fix`, `test`, `refactor`, `chore`, `docs`

The ticket number is always included. If a commit affects shared tooling with no ticket number,
use `chore(lab)` or `chore(setup)`.

Examples:
  feat(kspl-005): implement CorrelationContext AsyncLocal storage
  test(kspl-005): add unit tests for CorrelationMiddleware
  chore(kspl-005): mark ticket DONE

### Contract First

For every component, commit interfaces and data models before any provider implementation.
The sequence is always: contracts, then synthesizer, then provider, then unit tests, then
integration tests. Each is a separate commit. Each is pushed immediately.

### No Emojis, No Icons

All source code, comments, commit messages, documentation, and configuration files use plain
ASCII text only.

### No Comments That Describe What Code Does

Write a comment only when the WHY is non-obvious: a hidden constraint, a subtle invariant, a
workaround for a specific external behavior. Never comment on what a function does. Its name,
parameters, and return type communicate that. A comment that could be deleted without confusing
a future reader should be deleted.

### Accurate File Structure

Each ticket's Expected Outputs section lists the exact files to create and their exact paths.
Create exactly those files at exactly those paths. Do not rename, merge, or omit any file.

### Keep Ticket Status Current

When you begin a ticket: change its `Status` to `IN PROGRESS` in `implementation-plan.md`.
When all acceptance criteria are met and all tests pass: change it to `DONE`.
Both status updates are separate commits:

  chore(kspl-NNN): begin implementation
  chore(kspl-NNN): mark ticket DONE

## The Working Loop

Execute this sequence for every ticket, without variation:

1. Open `implementation-plan.md`. Identify the next `TO DO` ticket.
2. Verify all listed dependencies are `DONE`. If any are not, work on the blocker first.
3. Switch to the feature branch for this EPIC, or create it:
     git checkout -b feature/epic-NN-<phase-name>
4. Mark the ticket `IN PROGRESS`. Commit and push:
     chore(kspl-NNN): begin implementation
5. Read the `doc/` spec file named in the Pre-Implementation Requirement. Read it completely.
6. Commit and push the interface and data model files:
     feat(kspl-NNN): define contracts and data models
7. Commit and push the data synthesizer:
     test(kspl-NNN): add data synthesizer
8. Commit and push each provider implementation file individually:
     feat(kspl-NNN): implement <ProviderName>
9. Commit and push unit tests:
     test(kspl-NNN): add unit tests
10. Run the full test suite. Fix all failures before proceeding.
11. Commit and push integration tests:
      test(kspl-NNN): add integration tests
12. Run integration tests against Docker infrastructure. Fix all failures before proceeding.
13. Mark the ticket `DONE`. Commit and push:
      chore(kspl-NNN): mark ticket DONE
14. When all tickets in an EPIC are `DONE`, merge the feature branch to `main`:
      git checkout main
      git merge --no-ff feature/epic-NN-<name>
      git push

## The Progressive Integration Testbed

The `lab/` directory is not a test afterthought. It is a first-class deliverable that proves
emergent correctness — the property that components work correctly together, not just in
isolation. This cannot be verified by unit tests alone.

KSPL-026 (lab framework) must be completed after EPIC-02 and before EPIC-03.
Each canvas ticket (KSPL-027 through KSPL-033) must be completed before the next EPIC begins.
Canvas scenarios are run in Docker; they re-run all previous canvas assertions as regression.
When a canvas passes, `snapshot.py` creates and pushes a git tag (`canvas-N-stable`).

These tags are the stable baselines other developers use to adopt this library for new projects.
They must be correct. Do not create a canvas tag unless every assertion in the health report
passes.

## Philosophy

This library is built on one structural principle: a complex system gains robustness when its
components are loosely coupled at runtime and tightly coupled only through explicit, versioned
interfaces.

Every component exposes exactly one interface. Every implementation hides behind that interface.
The consuming service never knows which provider is running. The provider can change without the
consuming service changing a single line.

This means:

- Never inject a provider class. Always inject the interface.
- Never add a public method that is not defined on the interface.
- Never add configuration options that are not in the Options dataclass specified in the spec.
- Never implement features not described in the spec file. More capability is not better.
  It is technical debt that obscures the contract.

The full underwriting workbench capability emerges from the correct composition of 25 components.
No component knows about the workbench. Each component knows only its own contract. The
emergence is a consequence of precise interfaces, not of clever coupling between components.

Implement precisely what the spec says. No more. No less. The spec is the architecture.

## When You Are Uncertain

If the spec file and the implementation plan disagree on a detail: the spec file wins.

If the spec file is ambiguous on a detail: implement the simpler interpretation. Do not add
complexity to handle a case the spec does not mention.

If an acceptance criterion appears technically impossible as written: do not guess. Add an
"Implementation Notes" section to the ticket in `implementation-plan.md`, document the issue
clearly, implement the closest correct behavior, note the deviation, and push. The architect
will see it.

If a dependency is not yet `DONE`: do not start the dependent ticket. Return to the blocker.

If tests are failing and you cannot identify the root cause within a reasonable effort: add an
"Implementation Notes" entry to the ticket, describe exactly what is failing and what you have
tried, and push your current state. The architect will provide guidance.

Never push code that fails its acceptance criteria. When uncertain whether something meets the
criteria: it does not. Fix it first.
